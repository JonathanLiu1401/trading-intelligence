# digital-intern — onboarding & system reference

This is a **continuous financial news intelligence daemon**. It collects from ~17 sources in parallel,
scores everything locally with a GPU PyTorch model (`ArticleNet`), routes uncertain articles to Sonnet
4.6 for stronger labels, fires Bloomberg-style Discord alerts on urgent items, and emits an Opus 4.7
heartbeat briefing every 5 hours.

Its `articles.db` is the **read-only news source for the paper-trader** at `/home/zeph/paper-trader/`
(see `/home/zeph/paper-trader/CLAUDE.md`). Cross-system invariants live in both files; check the
**Backtest data isolation** section below before changing any query.

---

## 1. What digital-intern does (in one paragraph)

`daemon.py` spins up ~20 independent worker threads, each on its own polling interval. Collectors
(`collectors/`) pull articles from RSS feeds, GDELT, Reddit, web scrapers, SEC EDGAR, Google News,
Nitter, Substack, Finnhub, Alpha Vantage, Polygon, Massive, NewsAPI, Yahoo, Wikipedia. Every new
article is keyword-scored (`triage/heuristic_scorer.py`), inserted into `storage/article_store.py`
(SQLite, prefers a USB-mounted DB), then picked up by a scorer worker that runs `ArticleNet`
inference. Uncertain (`needs_llm=True`) articles are routed to Sonnet 4.6; high-urgency results
trigger an immediate alert via `watchers/alert_agent.py`. Every 5 hours an Opus 4.7 briefing is
generated by `analysis/claude_analyst.py` and posted to Discord. The model retrains every 2–3
minutes on accumulated labels. A Flask dashboard lives on `:8080`.

## 4. DB schema — `articles` table

```sql
CREATE TABLE articles (
    id          TEXT PRIMARY KEY,            -- sha256(url || title)[0:64]
    url         TEXT NOT NULL,
    title       TEXT NOT NULL,
    source      TEXT,                        -- e.g. "rss", "gdelt_2025-09", "backtest_run_42_winner"
    published   TEXT,                        -- RFC822-or-ISO; used by paper-trader backtest replay
    kw_score    REAL DEFAULT 0,              -- heuristic_scorer.py output, 0..5
    ai_score    REAL DEFAULT 0,              -- ArticleNet/Sonnet/Opus rolled-up score, 0..10
    urgency     INTEGER DEFAULT 0,           -- 0=normal 1=urgent 2=alerted
    full_text   BLOB,                        -- zlib-compressed
    first_seen  TEXT NOT NULL,               -- ISO timestamp at insert
    cycle       INTEGER DEFAULT 0,
    time_sensitivity REAL                    -- 0..1, ML-predicted, NULL until scored
);
CREATE INDEX idx_urgency    ON articles(urgency);
CREATE INDEX idx_ai_score   ON articles(ai_score DESC);
CREATE INDEX idx_first_seen ON articles(first_seen);
CREATE INDEX idx_cycle      ON articles(cycle);

CREATE TABLE briefings (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    ts              TEXT NOT NULL,
    text            TEXT NOT NULL,
    article_count   INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_briefings_ts ON briefings(ts);
```

### Key field semantics

- `id`: stable hash of `(url, title)`. Hash is computed in `article_store.article_id()`.
- `kw_score`: pure-Python keyword score, 0..5. Filled by `_ingest()` in `daemon.py`. Anything below
  0.5 is dropped before insert.
- `ai_score`: 0..10. Set by `score_pending()` after `ArticleNet` inference, or by `urgency_scorer`
  for grey-zone (Sonnet) articles.
- `urgency`: tri-state. `0` = normal, `1` = needs alert, `2` = alert already sent. Set by either
  scorer (if `urgency_pred >= 8.0`) or directly by Opus annotations.
- `time_sensitivity`: ML-predicted, 0..1. Used by `get_top_for_briefing` ranker to decay older
  articles intelligently. Bootstrap labels from `_time_sensitivity_label` regex in `trainer.py`.

## 5. Backtest data isolation — **critical invariant**

The article store contains **two classes of rows**:

1. **Live news** — from collectors. `url` is a real URL, `source` is something like `rss`,
   `gdelt`, `reddit`, `web`, `sec_edgar`, `google_news`, `finnhub`, etc.
2. **Synthetic/training data** — from paper-trader's backtest loop and Opus annotation pass.
   - URLs of the form `backtest://run_<id>/<sim_date>/<action>/<ticker>`
   - Source columns matching `backtest_*` (e.g. `backtest_run_42_winner`, `backtest_run_42_rank1`)
   - Source columns matching `opus_annotation*` (Opus's GOOD/BAD trade labels and lessons)
   - Always written with `urgency = 0`

**Every live read must filter out the synthetic rows.** The canonical filter is defined in
`storage/article_store.py`:

```python
_LIVE_ONLY_CLAUSE = (
    "url NOT LIKE 'backtest://%' "
    "AND source NOT LIKE 'backtest_%' "
    "AND source NOT LIKE 'opus_annotation%'"
)
```

It's applied in: `get_unscored`, `get_unalerted_urgent`, `get_top_for_briefing`. **If you add a new
query that feeds the live news pipeline, you MUST include this clause.** Paper-trader has the same
SQL fragment inlined in `paper_trader/signals.py` — keep them in sync.

Synthetic rows are intentionally kept in the same table because **ArticleNet trains on them**.
`ml/trainer.py::train()` pulls everything with `ai_score > 0` (no live filter) — backtest winners
and Opus lessons all contribute training signal. The split is "read for training: include backtest;
read for live alerts/briefings/inference targeting: exclude backtest."

## 8. ArticleNet — what it learns and how

PyTorch model (`ml/model.py`). Four-layer encoder with LayerNorm + GELU + Dropout, feeding four
heads. Trained multi-task:

```
loss = MSE(relevance, label_rel)                       # 0..10
     + 0.5 * BCE(urgency, label_rel >= 8.0)            # binary urgent
     + 0.2 * BCE(uncertainty, |rel_pred - rel_true|/10) # aleatoric proxy
     + 0.3 * BCE(time_sensitivity, label_ts)           # 0..1, heuristic bootstrap labels
```

At inference:

- **Production hot path** uses MC-Dropout (10 passes) to get `(rel_mean, rel_std, urg_mean, urg_std, ts_mean)`.
- `needs_llm = True` whenever the ensemble spread is wide (`UNCERTAINTY_REL` / `UNCERTAINTY_URG` thresholds in `ml/trainer.py`) or the prediction sits in the grey zone (`LLM_ZONE_MID_LO..LLM_ZONE_MID_HI`).
- `confident_noise = True` is set for clearly irrelevant items (`rel_mean < LLM_ZONE_CLEAR_NOISE`) so they're skipped entirely — no scoring, no alerting.

**Training data sources** (in priority/quality order):
1. Opus 4.7 heartbeat labels — strongest signal, ~50 articles per 5h.
2. Sonnet 4.6 urgency_scorer labels — moderate signal, every uncertain article.
3. Backtest winner injections — `backtest_run_*` rows with `ai_score=5.0` for BUY winners, `0.5` for SELL.
4. Opus annotation labels — `opus_annotation_cycle_*` rows tagged GOOD (5.0) / NEUTRAL (2.5) / BAD (0.5).
5. Heuristic `kw_score` — weakest signal, used only when nothing better exists.

## 10. Common failure modes

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| Multiple `daemon.py` processes spawning, high CPU | `heartbeat.sh` is fork-launching the daemon instead of pinging | `ps -ef \| grep daemon.py` to see the duplicates; review `heartbeat.sh` — it should `curl localhost:8080/health`, not `python3 daemon.py` |
| `database is locked` errors in logs | High writer contention during `purge_worker`'s `wal_checkpoint(TRUNCATE)` | Built-in retry already covers this (`_retry_on_lock`); if it still fails after 5 attempts something else holds the writer — check for stuck PRAGMA, lsof on the DB file |
| `score_pending` returning 0 in a loop | `_INFER_LOCK` held by another caller, or the model isn't fitted yet | Look for `[score_pending]` log lines — once fitted, batches drain to zero. If forever 0, run a manual retrain |
| Sonnet alerts not firing | `DISCORD_WEBHOOK_URL` missing in `.env`, or `mark_alerted` silently failed | `alert_agent.send_urgent_alert` returns False; check daemon.log; also confirm the URL is reachable |
| Heartbeat briefings missing | `analysis/claude_analyst.py::analyze` failed; daemon.log will show traceback | Often Opus quota or stale cache; tail the heartbeat worker section |
| GPU OOM on training | Concurrent backtest-side `_inject_and_train` colliding with `ml_trainer_worker` | Lower `RUNS_PER_CYCLE` in paper-trader's `run_continuous_backtests.py`; restart daemon to clear CUDA cache |
| Articles ingested but never alerted | `urgency` never set to 1 — model isn't predicting high urgency. Check `urgency_scorer` log lines | Re-train; lower `urg_mean >= 8.0` threshold in `score_pending` if persistent |
| Paper-trader live trader getting `backtest_run_*` articles in its prompt | Live filter not applied somewhere | Grep for `articles.db` in `/home/zeph/paper-trader/` and verify the `_LIVE_ONLY_CLAUSE` SQL pattern is present in every query. Check `paper_trader/signals.py` first |
| `articles.db` growing huge | `purge_worker` not running or backtest injection rate exceeds purge rate | Run `purge_old` manually; consider increasing `RETENTION_DAYS` is the wrong direction — lower it or add a per-source cap |
| `supervisor_state.json` shows half workers dead | A common dep (e.g., network) is down, or process was just restarted (workers re-ping every 5 min) | Wait one full `HEALTH_REPORT_INTERVAL_SECS=300`; check core workers (`rss`, `web`, `reddit`, `scorer`) first |
| Dashboard cross-fetch from paper-trader fails | CORS error in browser console | `dashboard.py::_cors` in paper-trader sets `Access-Control-Allow-Origin: *`; check both processes are up; verify `:8090` and `:8080` resolve from the browser |
| Backtest winner injections vanish from training pool | Some path is still using the **live** filter when it should be looking at *all* rows | `ml/trainer.py::train` should NOT include `_LIVE_ONLY_CLAUSE`; it deliberately reads everything with `ai_score > 0` |

| Add a per-article ML feature | `ml/features.py::extract_features_batch` plus `EXTRA_FEATURE_DIM`. Don't forget to bump `INPUT_DIM` accounting if you grow vocab |
| Add a new dashboard panel | `dashboard/` Flask app + a new JSON endpoint that reads from `articles.db` or `data/*.json` |
| Throttle a worker | Edit its interval constant at the top of `daemon.py` |
| Change retention | `RETENTION_DAYS` in `storage/article_store.py` |

## 13. Glossary

- **ArticleNet** — the GPU PyTorch model in `ml/model.py`. Don't confuse with paper-trader's
  `DecisionScorer` (a tiny CPU MLP).
- **Live-only clause** — the SQL filter that excludes `backtest://` URLs and `backtest_*` /
  `opus_annotation*` sources. Centralized as `_LIVE_ONLY_CLAUSE` in `storage/article_store.py`.
- **Grey zone** — relevance predictions where the model isn't sure (between
  `LLM_ZONE_MID_LO` and `LLM_ZONE_MID_HI`); these route to Sonnet via `needs_llm=True`.
- **Heartbeat briefing** — the 5h Opus 4.7 long-form market briefing posted to Discord.
- **Active learning queue** — `data/active_learning_queue.jsonl` of articles the model flagged
  as uncertain; capped at 5000 lines. `ml/recursive_labeler.py` reads this when picking what to
  re-label with Opus.
- **Supervisor** — the crash-tracking state machine in `daemon.py` (`_supervisor_lock` and
  friends). Decides whether a crashed worker respawns immediately, with backoff, or stays disabled.
